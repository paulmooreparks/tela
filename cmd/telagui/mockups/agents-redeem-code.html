Agent code redemption flow (proposed)

A new "Redeem Agent Code" path on the Agents tab, only useful when telad runs on the same machine as TV (developer workstations, single-host setups). Production agents on remote machines still redeem via telad pair on the agent host; this surface does not change that.

Two buttons sit side by side in the sidebar footer. Generate is primary because it is the more common operation; Redeem is secondary because it only matters for the local-agent-on-this-machine case.

The combined caption is a single sentence so the relationship is obvious: one direction generates, the other consumes.

The hub picker pulls from TV's known-hubs list. Picking "Other" reveals an inline URL input (not shown here for brevity). This handles the case where the operator was given a code for a hub they have not added yet.

The machine name is required even though it duplicates what the code already encodes. Reasoning: we want the operator to confirm what they expect, and the value must be passed to telad pair as a flag.

The result confirms what landed where, including the credstore path so the operator knows where the token lives. The telad service install hint is deliberate: pairing alone gets the token in place, but the agent will not actually run until telad is started (either manually or as a service).

Step 4 — Edge case: telad not installed locally
Bail out with a pointer to Updates

Redeem Agent Code

telad is not installed on this machine. Pairing requires a local telad binary; the Updates tab can install it for you.

If you are trying to register an agent running on a different machine, run telad pair there instead. Generating the code in TV does not deliver a token to the remote host; only the operator on the agent machine can complete the handshake.

The redeem flow checks for a local telad binary before showing the form. If none exists, the modal opens in this short bail-out state with a single action that routes the operator to the Updates tab where they can install it.

The second paragraph is a deliberate disclaimer: redemption in TV only works for local agents. The remote-agent case is explicitly called out so an operator who reached this dialog hoping to register a remote agent learns the right flow (run telad pair on that host).

Out of scope
What this flow does not do

Redemption in TV is local-only. TV cannot redeem a code on behalf of an agent on a different machine, because the resulting token has to land in that machine's credential store. The generate-side mockup (agents-pair-code.html) handles the remote- agent case correctly: it produces a code plus a copy-ready telad pair command for the operator to run on the target host.

TV also does not orchestrate post-registration startup. The result modal hints at telad service install but does not run it; service management is a separate workflow living in Client Settings (System Service card). A future iteration could fold a "Register and start as service" checkbox into the redeem modal, but only after watching how operators actually use this surface.
