Button vs badge design

The problem: in the current TelaVisor, an ONLINE status pill looks the same as a primary button (both are filled green rounded rectangles), the tb-btn secondary button looks identical to an outlined badge, and hovering is required to confirm clickability. This breaks down on touch screens and for users who do not want to hunt with the mouse. Color alone also fails for the ~8% of male users with red-green colorblindness.

The rule: A button always has elevation (subtle drop shadow) plus chrome (fill or border) plus cursor: pointer. A badge always has zero chrome (no border, no fill, no shadow) plus an icon prefix (dot or symbol) plus cursor: default. Neither rule depends on hover, and the two styles cannot be visually confused regardless of context.

1. The elevation invariant

Every button, regardless of variant, sits on a subtle drop shadow. This single signal tells the eye "raised object = pressable". Badges sit flat in the text flow. This works on first glance, without a mouse, and on any device.

7. Chrome buttons (topbar only)

The topbar icon strip (power, folder, alerts, info, settings, quit) is a distinct category. It cannot follow the elevation rule: shadows do not read well on the dark topbar background, and giving every icon its own shadow would make the strip look like a row of tile buttons — too heavy for chrome controls.

The solution is to treat the topbar as a known chrome region with its own convention. Users already recognize topbar icon strips from every desktop application (VS Code, Figma, browsers). Inside that region, the affordance rule changes: a persistent outlined circle with a meaning-carrying glyph is enough. The signals are:

  1. Persistent outline. The circle border is always visible, not only on hover. On a touch device you can see it before tapping.
  2. Repeated pattern. Same size, same shape, in a row. The repetition itself signals "these are all the same kind of thing".
  3. Meaning-carrying icon. Power, gear, X — glyphs that already carry action semantics. The glyph says "this does something" as clearly as elevation would.
  4. Distinct location. Chrome buttons only ever appear in the topbar. Content areas never contain them. The two sets never mix, so their visual styles never compete.

The rule is now four categories. Every interactive element in TelaVisor (and every TDL application) falls into exactly one:
  • .btn — content-area button. Rectangular, elevated, in the main view. Signal: elevation.
  • .status — content-area badge. Flat inline text, icon prefix. Non-interactive. Signal: flatness + glyph.
  • .chrome-btn — topbar icon button. Circular, persistent outline, glyph only. Signal: persistent outline.
  • .brand-link — topbar brand mark. Breaks the "visible non-hover affordance" rule to preserve branding strength. Signal: web convention + cursor + subtle hover brightness + focus outline.
Categories never mix locations. A .chrome-btn in content would be ambiguous; a .btn in the topbar would be visually loud; a .brand-link is a one-of-a-kind element that only ever exists at the top-left of the topbar. Location disjoint-ness is what keeps the four categories unambiguous.

About .brand-link

The topbar brand mark is the deliberate exception. Desktop UX rules say every interactive element must look interactive without hover, but the web has trained users for twenty years that a logo at the top-left is a clickable "home" link. For a design language that spans both a desktop app (TelaVisor) and a web app (Awan Saya), the web convention wins: users try clicking it, so the click must work.

Design choices for .brand-link:

  • Single color. The entire wordmark is accent green in both themes. No prefix/suffix two-tone split. One less case to think about when the theme flips, and it keeps the brand visually unified.
  • No underline on hover. The web convention says "link = underline on hover", but the brand mark is sacred and an underline fragments it. This is the single place where we break the convention.
  • Subtle brightness bump on hover (filter: brightness(1.15)). Enough to confirm interactivity to someone probing with the mouse, not enough to alter brand identity.
  • Focus-visible outline. A 2px accent outline with 4px offset on keyboard focus. Keyboard users must have a clear target since this is often the first Tab stop.
  • Per-app target. TDL does not dictate where a brand-link goes. Awan Saya: navigate to home page. TelaVisor: go to Status tab (the default landing view). Each app decides.

Alternatives considered

  • Move everything into a single ⋯ menu. Clean, but hides frequently-used actions (connect, quit). Two extra clicks to reach a primary workflow is too much.
  • Replace icons with labeled buttons ("Connect", "Quit", etc.). Defeats the chrome convention and steals space that content needs.
  • Use .btn-icon (the new small rectangular tile). Visually consistent with content buttons, but heavy in the topbar and creates ambiguity with content-area icon buttons like Refresh.
  • Keep as-is (current TelaVisor topbar). ← Chosen. Just formalize it with .chrome-btn and document the rule.

8. Accessibility summary

  • Touch: elevation is visible without hover. Buttons declare themselves on first glance.
  • Colorblindness: version status uses ✓/↑ glyphs so the meaning survives green/amber indistinguishability (deuteranopia and protanopia).
  • Focus: all buttons get a 2px --accent outline on :focus-visible for keyboard navigation. Badges are not in the tab order.
  • Screen readers: buttons are <button> elements, badges are <span>. The role difference is announced.
  • Contrast: primary button text is dark-on-green (#0b2016 on #2ecc71) which beats WCAG AA (contrast ratio 5.8:1). The old white-on-green scheme was 2.6:1 which fails AA.

9. Control reference

Canonical list of every interactive control class in TDL, what it looks like, where it lives, and when to use it. Treat this as the specification: if you need a control that doesn't fit any of these categories, stop and discuss it before inventing a new one.

.btn.btn-primary — Primary button
The main action in a view or modal. Exactly one per context. Filled accent green, elevated. Commits user intent (Save, Create, Connect, Apply).
  • Where: content area only; one per form, modal, or settings card.
  • Signals: elevation + accent fill + dark text + cursor: pointer.
  • Examples: Save (Application Settings), Connect (profile), Apply (hub config), Create (new token).
.btn — Secondary button
Default button. Any action that is not the primary commit. Elevated, outlined, neutral fill. Use for Cancel, Refresh, Logs, Browse, Restart, etc.
  • Where: content area, tab bars, toolbars, form footers.
  • Signals: elevation + neutral fill + border + cursor: pointer.
  • Examples: Cancel, Refresh, Logs, Browse, Restart, Check Now.
.btn.btn-danger — Danger button
Initiates a destructive action. Same geometry and elevation as a secondary button so it reads as "a button", with red text and a warning ⚠ glyph (injected via ::before) for color-independent identification. Hover commits to a filled red.
  • Where: content area, next to the item being acted on, or in a row of options.
  • Signals: elevation + red text + ⚠ glyph + cursor: pointer.
  • Examples: Delete token, Remove machine, Revoke access, Sign out.
  • Escalation: for destructive actions that cannot be undone (data loss), the danger button usually opens a confirmation modal that contains a .btn-destructive.
.btn.btn-destructive — Destructive button
The filled, high-commitment destructive action. Used only inside confirmation modals to finalize an irreversible action the user has already initiated with a danger button. The filled red background makes hesitation unambiguous and matches the weight of a primary button so the commit action is always the visually dominant choice.
  • Where: confirmation modals only, never in the main content area.
  • Signals: elevation + red fill + white text + ⚠ glyph + cursor: pointer.
  • Examples: "Delete permanently" (after clicking Delete in the Tokens view), "Discard changes" (after clicking Cancel with unsaved edits), "Remove machine forever".
  • Rule of one: a modal has at most one .btn-destructive. Its sibling is always a .btn labeled "Cancel".
.btn.btn-icon — Icon button
Compact square button with a single glyph. Same elevation as other content buttons. Used in dense toolbars or sidebars where a labeled button would be too large. Always carries a title tooltip for the action name.
  • Where: content area, in toolbars or list rows.
  • Signals: elevation + glyph + cursor: pointer.
  • Examples: sidebar refresh, up-one-level navigation, close dialog (content-area X).
  • Do not confuse with .chrome-btn: icon buttons are square and live in the content area; chrome buttons are circular and live in the topbar.
.chrome-btn — Chrome button
Circular icon-only button that lives in the topbar. Persistent outlined border is the affordance (no elevation). Reserved for application-level utilities: power, open data folder, alerts, about, settings, quit. Three color variants: neutral, accent, and danger.
  • Where: topbar only. Never in content.
  • Signals: persistent outlined circle + glyph + cursor: pointer.
  • Examples: Connect/disconnect, Open data folder, Alerts, About, Settings, Quit.
  • Topbar-scoped colors: chrome buttons read their foreground, border, background, and hover colors from --tb-chrome-* custom properties set by the containing topbar, so the same markup adapts to light and dark topbars without per-theme markup changes.
.brand-link — Brand link
Clickable brand mark in the top-left of the topbar. The deliberate exception to the "visible non-hover affordance" rule. Desktop UX would say this is wrong, but twenty years of web convention have trained users that a top-left logo goes home, so the click must work. Single .brand-link per application, max one per page.
  • Where: topbar, left side, always the first element.
  • Signals: cursor: pointer + subtle brightness hover (filter: brightness(1.15)) + 2px accent focus outline.
  • Styling: "Tela" prefix uses --tb-brand (flips with topbar theme), "Visor"/"Saya" suffix stays accent green in both themes. No underline, ever.
  • Target: decided per app. TelaVisor: Status tab. Awan Saya: home page.

.status — Status badge
Flat inline non-interactive label that communicates connection, health, or reachability state. Colored dot prefix (the primary signal) + uppercase label. No border, no fill, no shadow, no hover. Cannot be confused with a button.
  • Where: content area, inline with titles, sidebar items, list rows.
  • Signals: flat + colored dot + uppercase text + cursor: default.
  • Variants: .status-online (accent), .status-degraded (warn), .status-error (danger), .status-offline (muted).
  • Examples: Agent ONLINE badge, hub status, session state.

.status.status-current / .status.status-outdated — Version status
Specialization of the status badge for software version display. Monospace font, mixed case (so version strings render correctly), glyph prefix indicating update state. The glyph is the primary signal so users with red/green colorblindness can still distinguish "current" from "update available".
  • Where: anywhere an installed version is displayed.
  • Signals: ✓ (current, accent green) or ↑ (outdated, warn amber) + monospace version string.
  • Rule: applied to the installed version only. The available (latest) version is rendered in the default text color with no status decoration.
  • Examples: Installed Tools card (Client Settings), sidebar agent version, topbar TelaVisor version label, agent detail Version row.
Chip alternatives — pick one
The original .chip used a small outlined box. That style is historically interactive (GitHub labels, Stack Overflow tags, Steam tags), which makes the outlined pill a false signal for neutral metadata. Here are three candidates in a realistic sidebar-item context so you can pick the one that reads clearly as "metadata, not a control".

A. Outlined box (current, the problem)

Has a border. Borders around text mean "you can click this" in most web UI conventions. Even with cursor: default, the hand still hovers over it expecting something.

B. Flat filled pill

v0.7.0-dev.20

Prior art: Stripe Dashboard, Notion properties, Linear. Has presence, sits within the flow, reads as "painted label" rather than "interactive target". No border to suggest clickability, no elevation to suggest button-hood. The subtle fill (--surface-alt) gives it just enough weight to be noticed without competing with real controls.

C. Inline meta with middot separator

owlsnest-agent2 hubs
v0.7.0-dev.20

Prior art: macOS Finder info strip, VS Code status bar, git log oneline output, GitHub repo description headers. Pure text with a middot separator. Zero chrome. Absolutely cannot be mistaken for a control. Downside: less visually prominent, can blend in with the item it qualifies.

Recommendation

Adopt style B (flat filled pill). It preserves the visual scannability of a discrete tag while removing the interactive-looking border. It also composes naturally when you need more than one (e.g. "v0.7.0-dev.20" + "windows/amd64" + "2 hubs" side by side). Style C is cleaner but loses the ability to visually group metadata at a glance — the middot separators flatten them into an undifferentiated text run.

.chip — Neutral metadata chip
Small flat filled pill for neutral metadata tags. Muted text on a --surface-alt fill, no border, no elevation. Used for counts, platform tags, region labels. Explicitly not a state indicator — use .status for state.
  • Where: content area, inline with titles, list items.
  • Signals: flat filled pill + muted text + no border + cursor: default.
  • Why not outlined? Outlined boxes around text are a web convention for interactive tags (GitHub labels, Stack Overflow tags). A border is a false affordance for non-interactive metadata.
  • Examples: "2 hubs" next to an agent name, "windows/amd64" platform tag, region or environment labels.
